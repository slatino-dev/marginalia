# ARCHITECTURE: Marginalia

> Cloudflare-native agentic RAG. Written before structural code; updated as decisions land. Free-tier limits this project depends on are recorded below with their read date and re-verified at build start (they shift quarterly).

## System shape
One Cloudflare Worker (Hono + Zod at every boundary) serving `/api/*`, `/ask` (SSE), and the React SPA via Static Assets. D1 is the system of record (corpus, traces, evals, feedback). Vectorize holds chunk embeddings; chunk text and metadata live in D1, and Vectorize metadata carries only `{chunk_id, document_id, section}` for filtering. R2 holds source PDFs plus per-page extracted-text JSON for the audit panel's deep links. One Durable Object (SQLite-backed) is the neuron-budget ledger, single-flight coordinator, and holds the 00:00-UTC budget-reset alarm. Agent runs execute in the request context: LLM calls are I/O-bound awaits (which do not accrue CPU time), and Workers Free allows **10ms CPU per request** (developers.cloudflare.com/workers/platform/limits, read 2026-07-08 — same figure Conduit's charter verified this session). The synchronous CPU cost billed against those 10ms is Zod parsing + RRF fusion + citation-resolution + per-token SSE serialization; a **Phase 2 gate measures actual `cpuTime` on a real agentic `/ask`** (Workers Logs exposes it) before this shape is trusted. Named fallback if a real ask exceeds 10ms CPU: host the agent loop inside a Durable Object (separate, larger CPU allowance — confirm the current DO CPU limit at build start), which localizes the change to one module rather than the system shape. Cron Triggers run eval shards; the daily budget reset is a DO alarm (master plan prescribes DO alarms + cron). No Queues (paid, banned).

```
Visitor (SPA) ── /ask (SSE typed events)  ·  /api/* (REST, problem+json)
        ▼
Worker (Hono + Zod)
   ├─ AgentRun (request-context): planner(8b) → per-sub-query retrieval → sufficiency → synth(70b)
   │     ├─ Hybrid retrieval ──┬─ BM25 lane: D1 FTS5 (chunks_fts, external-content)
   │     │                     └─ dense lane: Vectorize topK (1024d or 768d)
   │     ├─ Reciprocal-Rank Fusion → bge-reranker-base (~40 cand) → top-8 context
   │     ├─ synthesis → inline citation markers → resolution validator
   │     └─ trace writer: every stage's candidates + scores → D1 (the product surface)
   ├─ BudgetLedger DO (SQLite): combined demo+eval daily counter + per-visitor counter, degrade-mode state, single-flight lock, 00:00-UTC reset alarm
   ├─ KV: preset full-trace cache (zero-neuron replay) — read-only, no locking (KV is eventually consistent; the DO is the only coordinator)
   └─ Cron: ONE nightly eval trigger → internal dispatcher (retrieval nightly · faithfulness shards on 4 assigned nights)
D1: documents · chunks · chunks_fts · queries · retrieval_steps · rerank_results · answers
    · eval_runs · eval_results · golden_questions · feedback
R2: source PDFs · per-page extracted text JSON        Vectorize: chunk embeddings
OFFLINE ingest (Node script, local/CI): fetch pinned PDFs → sha256 verify → per-page extract
    → structure-aware ~400-tok chunking (section path + page anchors + structural headers)
    → Workers AI REST embed → batch upsert Vectorize + D1 + R2
```

## Stack commitments
Hono + Zod (all boundaries) · D1 + drizzle-orm + FTS5 (versioned migrations in git; NEVER db push) · **dense lane = $0 brute-force cosine over D1-stored embeddings behind the `DenseIndex` port (2026-07-10 decision; NOT Vectorize — paid-gated; see Free-tier limits)** · Workers AI (bge-m3 or bge-base-en-v1.5 per the ingest gate; bge-reranker-base; llama-3.3-70b-fp8-fast synthesis; llama-3.1-8b planner/judge) · KV (preset trace cache, read-only) · one Durable Object (budget ledger + single-flight + reset alarm + resident embeddings for the cosine index) · R2 (sources) · Cron (evals, budget reset) · React SPA + shadcn/ui + AI Elements + Streamdown (streamed answer) + TanStack Table (eval tab) · promptfoo (eval harness) · PostHog + typed events.ts · Vitest + @cloudflare/vitest-pool-workers (miniflare D1/DO/KV are the real runtime; FTS5, RRF, cosine, and citation-resolution logic are all unit/integration-tested there). The dense lane is coded against the `DenseIndex` port (`src/retrieval/dense.ts`), so RRF fusion + rerank + citation resolution depend only on `DenseHit` and a future Vectorize/ANN backend is a drop-in swap of the implementation · structured console JSON + Workers Logs.

## Boundaries
- Handlers never touch D1 directly: repository modules only. The retrieval core lives in `retrieval/`, the agent loop in `agent/`, the eval harness in `evals/` (built as a clean liftable module for Crucible), the API layer in `api/`.
- Ingest is OFFLINE and out-of-band from the request path; the public Worker is strictly read-plus-ask and cannot write corpus rows.
- Retrieved corpus text and visitor questions entering LLM calls are UNTRUSTED data, never instructions (see Security posture).
- Vectorize and Workers AI are treated as fallible upstreams: every call has a timeout and a bounded retry; failures degrade, they do not 500.

## Invariants
- Every question -> one `queries` row + complete trace rows, written atomically via batched D1 writes.
- No answer renders a citation that fails resolution: unresolved markers are stripped and counted as `citation_resolve_failed`, surfaced honestly in the UI and in metrics.
- Neuron ledger is checked in the DO BEFORE any model call, debited after, and reconciled against the Cloudflare dashboard weekly.
- Preset cache entries are full stored traces, byte-identical to a live run's trace.
- The eval tab reads only committed `eval_runs` rows; no eval runs on demand.
- Corpus content is immutable post-ingest: re-ingest = a new document version, and old traces still resolve their chunk ids.
- Model id and prompt version are recorded on every answer and every eval run (`config_hash`), so the eval time-series survives model swaps honestly.
- All IDs are ULID; all times UTC ISO. No `latest` model pins (model ids are config, recorded per run).

## Error taxonomy
Typed classes -> RFC 9457 problem+json on the REST API; typed error events on the SSE stream. `RetrievalEmpty` / `BudgetExhausted` / `DegradedMode` / `UpstreamModelError` / `UpstreamModelTimeout` / `CitationResolveFailed` / `QueryTooLong` / `IngestChecksumMismatch` / `EvalRunOverlap`. Stack traces never leave the Worker; upstream (Workers AI / Vectorize) errors are wrapped, never passed raw to the client.

## Security posture (Opus-owned throughout)
- **Neuron-pool defense (the central risk).** The 10,000 neurons/day Workers AI allocation is account-level and SHARED with the other portfolio apps. Marginalia's ceiling is set by the portfolio neuron-allocation table (PORTFOLIO-V2.md): **4,000/day, a SINGLE combined ceiling covering demo asks AND eval runs** — not a demo share with evals tracked separately (separate tracking does not create neurons and would let eval nights push Marginalia's true draw past its share and starve siblings). The BudgetLedger DO holds one daily counter debited by every model call from any source (demo, retrieval eval, faithfulness shard), plus a per-visitor daily cap, both checked before the call. Eval is reserved first each day (it is the product thesis and runs at lowest-traffic hours); the demo consumes the remainder. When the remainder is exhausted the degrade ladder engages (full agentic -> single-pass -> retrieval-only ~5 neurons/question -> cached presets only); the demo never goes dark and no single visitor or bot can drain the shared pool. On the four faithfulness-shard nights the reserved eval slice is larger, so the demo's remainder is smaller and the ladder may engage one rung earlier that day — stated honestly in the degrade banner, not hidden (see docs/eval-spec.md scheduling). Visitor identity for the per-visitor cap is a salted, rotating, non-PII session token, never an IP stored raw — and because that key rotates, a client can reset its per-visitor counter by forcing rotation, so the per-visitor cap is **best-effort fairness only**. The authoritative shared-pool protections are the global combined 4,000/day ceiling and the per-IP-hash sliding-window limit (below), neither of which a rotating session key can evade; the neuron-pool defense does not rest on the evadable per-visitor key.
- **Per-call output cap.** Every Workers AI call (70b synthesis, 8b planner, 8b judge) carries an enforced `max_tokens` ceiling, independent of the aggregate daily/per-request budget, so a single runaway generation cannot blow the budget or the latency guardrail. The synthesis cap is sized to the reading-column measure; planner/judge caps are sized to their structured outputs.
- **Rate limiting on every public route, hardest on `/ask`.** Sliding-window counters in the DO, per-session AND per-IP-hash, returning `RateLimit-*` headers and `429` + `Retry-After`. `/ask` additionally carries a per-request token/neuron budget, not just a request count, per the AI-endpoint rule.
- **Prompt-injection posture.** Visitor questions and retrieved corpus passages are wrapped as clearly delimited untrusted data in every prompt; system instructions live only in the trusted preamble; the planner and synthesis prompts state that retrieved content is evidence to cite, never commands to follow. The synthesis output is parsed for citation markers only; no tool-execution authority is granted from model output on the public path. Question length is capped (`QueryTooLong`) to bound injection surface and cost.
- **Admin/ingest isolation.** Corpus writes, feedback moderation, and eval management live behind an authenticated admin surface (server-side session, HttpOnly/Secure/SameSite cookie; argon2id-hashed credential in a Worker secret; uniform auth errors, no user enumeration; hardest rate limits on the login route). The admin surface is unreachable from the public demo Worker routes; the public surface exposes read-plus-ask only. Every privileged admin action (trace redaction, feedback moderation, ingest run, eval-config change) writes an append-only D1 audit row (`admin_audit`: actor, action, target_id, timestamp, reason) — a who/what/when record on anything that mutates or destroys data, per the backend audit-log rule.
- **Output rendering / transport hardening.** The answer view and audit panel render LLM output, corpus passages, the visitor's own question text, and the planner-generated sub-queries into the DOM — and the public share-permalink/trace view echoes those visitor-supplied and planner-derived strings to third parties, so they are a stored-XSS surface, not just the model output. ALL of that text (model output, corpus passages, question text, sub-query text) renders as sanitized markdown with NO raw HTML (Streamdown, never `dangerouslySetInnerHTML`), so neither a passage, a completion, nor a crafted question can inject script; CSP is defense-in-depth behind that. SPA/HTML and API responses carry a Content-Security-Policy plus standard security headers (X-Content-Type-Options, Referrer-Policy, frame-ancestors none); request bodies have a size limit; CORS is same-origin only for the SPA/API (the v1.x MCP surface gets its own explicit allow-list). CORS is never `*` with credentials.
- **Ingest SSRF and supply-chain control.** The offline ingest fetcher accepts only the pinned, allow-listed source URLs recorded in docs/corpus-dossier.md; each download is sha256-verified against the dossier before extraction (`IngestChecksumMismatch` aborts). No model-supplied or visitor-supplied URL is ever fetched.
- **Feedback endpoint.** The public "flag this citation" endpoint is rate-limited per session, accepts only an enum verdict + a length-capped note (Zod-validated), captures NO PII, and is write-only into a moderation queue read only by admin. Abuse content in notes is flagged for admin review, never rendered publicly.
- **Secrets.** PostHog key, Sentry DSN, and admin credential live in Worker secrets; nothing sensitive is exposed client-side (PostHog uses its public project key only). No secret material in any log or response.
- **Analytics and trace retention.** Tracking events carry no PII and no raw question text (enforced in the Zod events schema); question text lives only in D1 traces. D1 question-text retention has a documented abuse-content handling policy (admin can redact a trace; redaction preserves the row for eval integrity while blanking the text).
- **v1.x MCP endpoint.** When the MCP surface ships it authenticates consistently with Patchbay's endpoint-auth pattern (scoped bearer key, same rate limits and audit as the REST surface) before Marginalia registers as a tenant.
- A full `security-review` skill pass gates Phase 3 before launch, covering the ask path, admin surface, feedback endpoint, and eval cron.

## Free-tier limits this project depends on
> **Build-start re-verification (2026-07-10, Marginalia Phase 0).** Every limit below was re-read this session from developers.cloudflare.com/workers/platform/pricing and .../workers/platform/limits. All free daily limits reset 00:00 UTC and fail with an error when exceeded (not silent). One material change surfaced since the 2026-07-08 charter read: **Vectorize now requires the Workers PAID plan** — see the flagged finding below; it is a Phase 1 blocker, not a Phase 0 one.

- **[RESOLVED 2026-07-10 — dense lane is $0 brute-force cosine, NOT Vectorize] Vectorize** (developers.cloudflare.com/workers/platform/pricing, read 2026-07-10): the pricing page's Vectorize section now leads with "**Vectorize is currently only available on the Workers paid plan**", above a table that still lists a Workers Free allowance (30,000,000 queried / 5,000,000 stored vector dimensions/month). Read together, the free *allowance* is only usable once the account is on the Workers **Paid** ($5/mo) plan, which collides with the portfolio's $0/free-tier posture (PORTFOLIO-V2). **Decision (team-lead-approved 2026-07-10): option (b) — the dense lane is implemented as $0 brute-force cosine over the corpus embeddings, NOT Vectorize.** It sits behind the `DenseIndex` port (`src/retrieval/dense.ts`) so a future Vectorize/ANN backend is a drop-in (same interface); RRF fusion + rerank + citation resolution depend only on `DenseHit`. **Measured this session** (`npx tsx scripts/bench-dense.ts`, Node = identical V8 float math; workerd coarsens `performance.now()` so it cannot be timed in-runtime): a full top-8 cosine scan over **1,900 x 1024 costs ~1.15 ms/sub-query and ~3.4 ms for a 3-sub-query agentic question** — under the 10ms Worker CPU budget, and the `CosineIndex` lives in the same Durable Object as the agent loop (System-shape DO fallback), so it draws on the DO's 13,000 GB-s/day duration allowance rather than the 10ms Worker limit. Consequence: **the 5M-stored-dim Vectorize cap NO LONGER binds corpus size** — the sizing gate is now only about embed rate + quality, and the embeddings live in D1 (loaded into the DO once). Vectorize figures retained only for the drop-in option: max 1536 dims/vector, topK <=50 with metadata, 10KiB metadata/vector. **Portfolio note: PORTFOLIO-V2.md may need a portfolio-level update since other charters may assume Vectorize — the team lead owns that file.**
- **Workers AI** (rates verified 2026-07-08; re-confirm neuron rates at Phase 1 ingest): 10,000 neurons/day, **account-level and shared across all six portfolio apps**. Marginalia's slice is the 4,000/day combined ceiling set in PORTFOLIO-V2.md's shared neuron-allocation table — never the raw 10,000. Rates: bge-m3 1,075 n/M-tok (1024d), bge-base-en-v1.5 6,058 n/M (768d), bge-reranker-base 283 n/M, llama-3.3-70b-fp8-fast 26,668 in + 204,805 out n/M, llama-3.1-8b 25,608 in + 75,147 out n/M.
- **D1** (verified 2026-07-10, developers.cloudflare.com/workers/platform/pricing): Free plan **5,000,000 rows read/day, 100,000 rows written/day, 5 GB total storage** (summed across ALL D1 databases on the account — shared with the sibling apps, so this is a coordination point). Indexes add one written row per indexed column on write; DDL mixes read+write rows. D1 export does not support virtual tables, so the runbook documents drop-FTS-then-export.
- **Workers KV** (verified 2026-07-10): Free plan **100,000 keys read/day, 1,000 keys written/day, 1,000 deleted/day, 1,000 list requests/day, 1 GB stored**. The 1,000-writes/day cap is the binding one — the preset-trace cache is read-heavy and write-rare by design (ARCHITECTURE invariant: KV read-only on the request path, populated offline/rarely).
- **Workers platform** (verified 2026-07-10, developers.cloudflare.com/workers/platform/limits): Free plan **100,000 requests/day** (Error 1027 when exceeded; requests to static assets are free and unlimited), **10 ms CPU per HTTP request** and **10 ms CPU per Cron Trigger** (network waits do NOT count — LLM/Vectorize awaits are free of CPU time; this is the System-shape gate figure and the DO-fallback trigger), **50 subrequests/request** (a real ceiling on the agent loop's fan-out: each Vectorize query, Workers AI call, and D1 batch is a subrequest — Phase 2 must keep an agentic `/ask` under 50), **6 simultaneous outbound connections/request**, Worker size 3 MB, startup 1 s, **5 Cron Triggers/account** (load-bearing: Marginalia uses ONE cron + a DO alarm for the budget reset), 20,000 static-asset files/version at 25 MiB each. Request body max 100 MB (Free CF plan) — Marginalia caps far tighter in app code.
- **Durable Objects** (verified 2026-07-10, developers.cloudflare.com/durable-objects/platform/pricing): Free plan **SQLite-backed classes ONLY** (KV-backed DOs are paid — the BudgetLedger DO must use `new_sqlite_classes`), **100,000 requests/day** (HTTP + each RPC session + WebSocket message + alarm invocation counts as one), **13,000 GB-s/day compute duration**. SQLite storage is NOT billed on Free. Idle+hibernatable DOs are not billed for duration. This is the allowance the agent-loop-in-DO fallback (System shape) would draw on if a real `/ask` exceeds the 10ms Worker CPU budget.
- **R2** (verified 2026-07-10): Free **10 GB-month storage, 1,000,000 Class A ops/month, 10,000,000 Class B ops/month, egress free**. Source PDFs + per-page extracted-text JSON fit comfortably (the seed corpus is a handful of PDFs, tens of MB).
- **Workers Logs** (verified 2026-07-10): Free **200,000 log events written/day, 3-day retention** — relevant to the Phase 2 `cpuTime` measurement gate (Workers Logs exposes cpuTime).

## Free-allocation math (recompute at build start; rates verified 2026-07-08)
- **Vectorize storage** (corpus-size constraint): 5M stored dims / 1024d (bge-m3) = 4,882 vectors max; / 768d (bge-base) = 6,510. Rule: prefer bge-m3 (best quality, cheapest embed rate) if the measured chunk count <=4,400 (90% of cap); else fall back to bge-base 768d. Planning estimate ONLY (unverified — actual page counts measured at ingest, see corpus-dossier.md): the corrected ~1,000-1,100 page estimate at ~400-tok chunks projects to ~1,900 chunks (central case), with ~4,400 as the unverified worst-case bound. If the measured count lands at the central estimate, bge-m3 is the choice and the model-tension never arises; the sizing gate still decides from the measured count.
- **Vectorize query budget**: the ~4,400-vector resident corpus at 1024d = ~4.5M stored dims billed against the 30M queried pool EVERY month (the recurring stored term, not one-time inserts), leaving ~25.5M for actual queries. So ~25.5M / 1024 ~= 24,900 queries/month; at ~3 sub-query retrievals per agentic question ~= 8,300 questions/month ~= 275/day. Not the bottleneck; the neuron ceiling is.
- **Workers AI neurons** (the binding traffic constraint, sized against the 4,000/day combined ceiling): per-question cost is a function of retrieval steps under the max-4-step budget. v0.5 single-pass ~= 280 neurons (5k-tok context x 26.7 n/k input + 700 output tok x 204.8 n/k on 70b, +~4 rerank of 40 candidates, +<1 query embed). v1 agentic adds the loop overhead on top of the one final 70b synth (~280): **typical 2 steps ~= 370** (280 synth + ~53 8b planner + 2×[~4 rerank + ~15 8b sufficiency check + <1 embed] ~= 40); **worst-case 4 steps ~= 450** (larger fused context into synth + 4×[rerank + sufficiency + embed] ~= 80). So the demo's slice of the ceiling sustains, per day: on a non-shard night evals reserve ~300 → ~3,700 remainder ≈ **~10 typical / ~8 worst-case novel agentic questions**; on a faithfulness-shard night evals reserve ~2,475 → ~1,525 remainder ≈ **~4 typical questions** before the ladder engages. Plus unlimited zero-neuron cached preset replays and ~300/day retrieval-only queries (~5 neurons each) at the low rungs. One-time ingest embedding: at the corrected ~1,900-chunk central estimate ~= 0.8M tok x 1,075 n/M ~= 850 neurons; the ~1.9M-tok / ~2,000-neuron figure is the worst-case ~4,400-chunk bound, not a point estimate. Drawn once, coordinated as a one-off against the pool.
- **Eval budget** (inside the 4,000 combined ceiling, reserved first): nightly retrieval metrics on 60 questions ~= 300 neurons (embeds + rerank, no generation). Faithfulness on the 20-question sentinel subset **sharded 5/night across 4 assigned nights** (not 10/night over 2): 5 × ~435 (single-pass pipeline + 8b judge) ~= **2,175/night**, run 03:00-05:00 UTC. So a shard night draws ~2,475 total (retrieval + faithfulness) and leaves ~1,525 for the demo (ladder may drop one rung — accepted, banner-stated); non-shard nights draw ~300 and leave ~3,700. Full-60 faithfulness only on release candidates, and only if the day's remaining ceiling allows (else deferred, never paid).

## Data model (D1, drizzle-orm, versioned migrations)
`documents` (ULID, title, source_url, sha256, pages, license_note) · `chunks` (id, document_id, seq, section_path, page_start/end, text, token_count) + `chunks_fts` (FTS5 external-content) · `queries` (id, session_id, question, mode agentic|single|retrieval_only|cached, plan_json, latency_ms, neurons_est, outcome) · `retrieval_steps` (id, query_id, step_n, sub_query, lane bm25|dense, rank, chunk_id, score) · `rerank_results` (query_id, chunk_id, score, kept) · `answers` (id, query_id, text, model, config_hash, citations_json, ttft_ms) · `eval_runs` (id, ran_at, suite retrieval|faithfulness|ablation, config_hash, metrics_json) · `eval_results` (run_id, question_id, per-question scores) · `golden_questions` (id, question, expected_chunk_ids, expected_answer_notes) · `feedback` (id, query_id, marker, verdict, note) · `admin_audit` (id, actor, action, target_id, ts, reason; append-only, covers trace redaction / feedback moderation / ingest runs / eval-config changes). No raw string SQL; FTS5 MATCH via parameterized queries only.

## Runbook
`npm run dev` (wrangler dev; **local D1/DO/KV + REMOTE Vectorize binding** — Vectorize has no local simulator, so the dense lane binds to a real remote index via wrangler's remote-binding flag; the rest of the stack stays local) · `npm test` (vitest-pool-workers) · `npm run migrate` (drizzle-kit generate + wrangler d1 migrations apply) · `npm run ingest` (offline Node pipeline; requires admin creds; writes Vectorize + D1 + R2) · `npm run evals` (promptfoo suite; also the cron entrypoints) · `npm run deploy` (CI-only on main; local deploy blocked). **D1 export caveat**: virtual tables are unsupported by export, so back up with `DROP chunks_fts` -> export -> rebuild FTS from `chunks`.

## Anti-list
No Queues (paid) · no user uploads v1 · no raw string SQL (drizzle + bound params; FTS5 MATCH parameterized) · no unversioned prompts (prompt version in `config_hash`) · no eval number on the tab without a persisted `eval_run` row · no `latest` model pins · no dashboard/eval feature before its API contract test exists · no PII or raw question text in analytics · no visitor- or model-supplied URL ever fetched by the Worker or the ingest script.
